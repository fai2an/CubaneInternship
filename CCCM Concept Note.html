CCCM Concept Note

Introduction:

Cubane is a SaaS-focused Layer-1 blockchain designed to solve real-world challenges in industries like digital infrastructure and digital products. Cubane uses a custom built consensus named Cubane Cubic Consensus Mechanism[CCCM], where it focuses on privacy, scalability and consistency. CCCM operates on a hybrid modular architecture, leveraging zkProofs and FHE for data privacy, PoET and HTLC over on-chain timestamping and Cubic mechanism with POS over Nodes for Validation.
CCCM can be classified into 4 modular layers that operate independently. We can get a simplified understanding of CCCM from understanding the transaction flow in CCCM.

Transaction node:

Initiation of transaction requests and compute usage for validation and scaling.
Transaction node initiates a transaction request and goes through a quick check via Dynamic Resource Allocation Layer [DRAL] to manage CPU usage. DRAL manages the computation power allocation based on transaction complexity and CPU stats.
DRAL allocates x% of compute of total 100% Transaction Node and throws back a “validation request” for validation of transaction towards the Tx node. The “Transaction Node” validates the transaction and sends the “Transaction Request” ahead. Before going On-chain, the transaction goes into a ZK Layer.
The rest 100% - x% of the transaction node reacts as the Member Node in Cube [MNCU], scaling the computation power with the network.

ZK Layer:

Generation of Zk-proofs [zkSTARKs] and batch processing of transactions for Onchain verification.
All transaction requests once validated on Transaction Node are sent over to the ZK Layer. ZK Layer accepts these “Transaction Requests” and generates ZK-proofs [zkSTARKs] of these transactions. All such “ZK-proofs” go through the “PoET + HTLC” layer and are “Time Stamped On-chain”. This ensures proper sequencing and consistency of transactions.
ZK-proofs are Time stamped and are ready to be processed for verification. Now, these ZK-proofs are batch processed and are sent through a “Randomization Algorithm” and are sent into Random Shard.

Validation Layer:

Transaction validation in CCCM via cubes and POS.
Shards receive ZK-proofs of transactions along with the Timestamps and are modeled to process further. Each shard runs a “Randomization Algorithm” and “Cube Organizer Algorithm” to organize multiple cubes. These cubes are assigned with different sets of “ZK-proofs”.
Each Cube consists of a Leader node [LDN] and multiple Member nodes[MN], the Leader node is selected via “Randomization Algorithm” and  “POS”, it is also responsible for the Cube networking and syncing mechanism.
Leader Node verifies the transaction and its metadata along with performing “Tick Algorithm” on time-stamp [provided by “PoET + HTLC”]. Once the transaction is verified by the Leader node, it generates an output and broadcasts it to the Member Nodes of the cube.
Member Nodes [MN] of the cube are responsible for validating the output sent by the Leader Node. This process uses the POS [Proof of stake] mechanism and once the majority is reached, the transaction is considered to be validated.
Validated transaction is now ready to embed On-chain and is sent to “ZK-Compression Layer”.
